# Archived decisions — paper sync + watch-folder (retired 2026-05-07)

These four entries describe code and actions that were superseded by the Mendeley-as-truth
pivot (2026-05-01) and permanently retired in Steps 11.9 and 11.R. Archived from
`decisions.md` on 2026-05-07. The durable record is intact; nothing was summarised.

---

## 2026-04-25 — Skip watch-folder cleanup in `sync-mendeley` (deviation from skill)
**Context:** The paper-pipeline skill copies `<id>.pdf` from `database_dir` → `watch_dir` in `sync promote`, lets Mendeley auto-import (which leaves the Watch copy in place), then in `sync mendeley` deletes the Watch copy *after* confirming `mendeley_id` is populated. This requires a guardrail block ("Never delete Watch copy without confirmed `mendeley_id`. Never touch Database original.") and introduces a race window between Mendeley import and deletion confirmation.
**Decision:** Docent's `sync-promote` will copy DB → Watch and stop there. `sync-mendeley` will be a pure read/cross-check op that populates `mendeley_id` from Mendeley's library; it will NOT delete the Watch copy. Watch accumulates files over time. If accumulation becomes annoying, add a separate on-demand `paper clean-watch` action later (designed only if forced).
**Why:** Mendeley dedups on import (DOI/title), so re-promoting a file is safe-but-pointless; not destructive. Removing the cleanup eliminates: (a) the guardrail, (b) the race between "Mendeley imported" and "we confirmed it before deleting", (c) the only reason `sync-mendeley` needs filesystem write access. Cost is monotonic Watch growth — at ~3MB/PDF and a few hundred/year, <1GB. Mostly nothing.
**Alternatives rejected:** Port the cleanup verbatim — adds the most fragile code in the skill for ~no real win. Cleanup based on a TTL ("delete after 7 days") — implicit time-based deletion of user files is hostile; explicit `paper clean-watch` is honest.
**Status:** Superseded — `sync-mendeley` and `sync-promote` both retired in Step 11.9. Watch-folder concern is moot (database_dir IS the watch folder now, no separate subdir).

## 2026-05-01 — paper.py carve-out (paper_metadata.py + paper_sync.py)
**Context:** After Step 11.2 (sync-pull + bug batch + identifier-free drop), `paper.py` was 1253 LOC and would grow further with Step 11.3 (`sync-promote`). The metadata fallback chain (DOI → CrossRef → PDF info → filename) and the curl-based Unpaywall/download helpers were both cohesive groups that didn't need access to PaperPipeline state.
**Decision:**
1. **Split criterion: pure-vs-stateful, not by Step number.** Helpers that are pure functions of (inputs, executor) → `paper_metadata.py`. Curl-based sync helpers → `paper_sync.py`. Methods that touch `self._store` / `self._log_event` (action methods, `_sync_pull_run`, `_resolve_scan_folder`) stay on `PaperPipeline`.
2. **Models stay in `paper.py`.** Consumed in `paper_metadata.resolve_metadata` via `TYPE_CHECKING` import to avoid a circular runtime dep. Moving models to `paper_models.py` adds a third file with no behavior — defer until justified.
3. **Tests patch the new module paths**, not `PaperPipeline` class attrs. Keeps the seam honest (mocks point at where the function actually lives) and avoids carrying shim methods on the class.
**Why:** (Pure-vs-stateful) coupling-reducing rather than coupling-shifting — `_sync_pull_run` would have needed store + log injection to move out, which is more wiring than the carve-out earns. (Models stay) the import direction is paper.py → paper_metadata.py; flipping it via `paper_models.py` is symmetric churn. (Honest mocks) class-attr patches were a vestige of the helpers having been instance methods; module-level functions deserve module-level patches.
**Alternatives rejected:** Move models to `paper_models.py` (extra file, no behavior). Keep shim methods on `PaperPipeline` so test patches still work (defeats the carve-out). Move `_sync_pull_run` too (would require threading `store` + `log` through, increasing surface area). Do the split only after Step 11.3 lands, when both `sync-pull` and `sync-promote` exist (defensible — could yield a better seam — but pre-split costs nothing and gives 11.3 a clean home).
**Status:** Superseded — `paper.py`, `paper_metadata.py`, `paper_sync.py` all deleted in Steps 11.8/11.R. The pure-vs-stateful split principle carried forward as `reading.py` / `reading_store.py` / `mendeley_client.py` in `bundled_plugins/reading/`.

## 2026-05-01 — Step 11.3 `sync-promote` chooses move over copy
**Context:** The original Step 11 plan (`paper_pipeline_port_plan.md`) and the 2026-04-25 watch-cleanup decision both assumed `sync-promote` would *copy* DB → Watch and leave the DB original in place, with Mendeley auto-importing from Watch. The user pushed back during 11.3 design: copy means two on-disk PDFs per paper (DB + Watch) until Mendeley imports, then three counting Mendeley's own library. They wanted no duplicate PDFs lying around.
**Decision:**
1. **Move, not copy.** `paper_sync.move_to_watch()` uses `shutil.move`; the DB original is gone after promotion. Watch holds the only filesystem copy until Mendeley imports it into its own library.
2. **New `QueueEntry.promoted_at: datetime | None` field.** Set on successful move (or heal). `sync-status.promotable` now ANDs `promoted_at is None` with "filename not in Watch" — the Watch-filename check alone (Step 11.1's stand-in) doesn't survive move semantics because a healed entry might already sit in Watch from a manual move.
3. **Two heal branches** for inconsistent prior state: (a) `pdf_path` already inside Watch → set `promoted_at`, no move; (b) `pdf_path` rotted but a file with the expected filename exists in Watch → repoint `pdf_path` and set `promoted_at`. Both report under a `healed` bucket distinct from `promoted`.
4. **Collision = hard fail, not overwrite.** `move_to_watch` raises if the target filename already exists in Watch and isn't the same inode. Caught by the action and reported under `failed` with the conflicting path.
5. **Buckets:** `promoted` / `already_promoted` / `healed` / `missing_file` / `not_eligible` / `failed` / `dry_run_promote`. Generator action; `--id` / auto-mode / `--dry-run`.
**Why:** (Move over copy) the user owns the storage; "no duplicates" is a real preference, not a micro-optimization. Mendeley import is one-way for our purposes — once it has the file, the Watch copy is the *only* recovery path if Mendeley misbehaves, which is a feature, not a bug. (Heal branches over "refuse to act") users will inevitably move files manually; an action that bails on inconsistent state forces them into manual queue surgery. Healing is cheap and observable. (Hard-fail on collision) silent overwrite of an existing Watch file would risk clobbering an in-flight Mendeley import. (`promoted_at` as truth, not derived) Watch-filename match is a proxy that lies after manual moves; an explicit timestamp is honest.
**Alternatives rejected:** Copy with cleanup in `sync-mendeley` (the skill's original design — fragile guardrails, race window, contradicts the no-duplicates preference). Copy and leave forever (the Step 11.1-era plan — duplicates accumulate, contradicts the preference). Move + auto-rename on collision (silent rename hides real bugs; Watch filename is the join key sync-status uses). Defer `promoted_at` to a later step (Step 11.1 already deferred it; pushing further would mean `sync-status.promotable` keeps lying about healed entries).
**Status:** Superseded — `sync-promote` action retired in Step 11.9. `database_dir` IS the watch folder (no separate promotion step needed). `QueueEntry.promoted_at` field removed in Step 11.10.

## 2026-05-01 — Step 11.4 sync-mendeley: in-process MCP client over executor-subprocess
**Context:** `sync-mendeley` needs to call the Mendeley MCP server (`pallaprolus/mendeley-mcp`, launched as `uvx mendeley-mcp`) for two tools: `mendeley_get_by_doi` and `mendeley_search_library`. Two wiring shapes were on the table — reuse `Context.executor` to spawn the server and speak JSON-RPC by hand, or pull in the official `mcp` Python SDK and use `ClientSession` + `stdio_client`. The executor seam carried Step 11.2's whole sync-pull suite via FakeExecutor pattern-matching curl args, so the precedent existed.
**Decision:**
1. **In-process MCP client via the official `mcp` SDK** (`mcp>=1.0,<2`, lazy-imported inside `mendeley_client.py` so `paper.py` import stays cheap). Function-level wrappers `lookup_doi(doi, launch_command)` and `search_library(query, launch_command, limit)` return `{"items": list, "error": str | None}`. Errors are prefix-classified (`auth:` / `transport:` / `tool:`) so the action can route to `failed` and append a `mendeley-auth login` hint when warranted.
2. **Spawn-per-call,** not session-pooled. Real-data clocked ~5.3s end-to-end per call (subprocess + OAuth token refresh + Mendeley GET); pooling would save the subprocess slice only.
3. **Tests monkeypatch the two module functions at the import site** (`docent.tools.paper.mendeley_lookup_doi` / `mendeley_search_library`) — same shape as the 2026-05-01 carve-out decision said to use for `paper_metadata` and `paper_sync`. No `Context`-level injection slot for the Mendeley client.
4. **`mendeley_id` is whatever the response carries** — `id` (UUID) for library documents from search, `catalog_id` for catalog items from `mendeley_get_by_doi`. `_extract_mendeley_id` checks both. Caught during real-data validation: catalog responses lack `id`, which would have silently bucketed every DOI hit as `not_found` ("match returned but no Mendeley id field").
5. **`PaperSettings.mendeley_mcp_command: list[str] | None = None`** added to settings; default lives in the wrapper (`["uvx", "mendeley-mcp"]`). Not yet exposed via `config-set` (list-typed; `config-set` only handles strings today). Power users edit `config.toml` directly.
6. **Auto mode targets `promoted_at is not None` AND `mendeley_id is None`.** `--id` override bypasses both filters. Search returns >1 result → bucketed as `ambiguous` with up to 5 candidate summaries; user must `--id` + edit to disambiguate. No fuzzy threshold yet — wait for real-data signal (Step 11.2 only added the title-match guard after a real Arabic-paper bug).
**Why:** (SDK over hand-rolled JSON-RPC) MCP is stateful (handshake → list_tools → call_tool); the executor seam is shaped for one-shot CLI calls like curl. Reinventing framing + lifecycle by hand was strictly worse. (Spawn-per-call) sync-mendeley runs at most a few dozen entries; ~5s cost is dominated by network and OAuth, not subprocess spawn — pooling would save little. (Function-level seam over class) parallels `paper_sync.unpaywall_lookup` shape exactly; one less abstraction. (Both id keys) the surprise was real-data finding; defending against it costs one line. (No fuzzy threshold) speculation; let the data speak first.
**Alternatives rejected:** Subprocess via `Context.executor` (would have meant hand-rolling JSON-RPC framing — strictly worse than the SDK). `MendeleyMCPClient` instance class (no state worth carrying between calls; the ctor would just hold `launch_command` which the function arg already does). Inject the client through `Context` like `executor`/`llm` (no second consumer; speculative). Pool MCP sessions across the run (premature; 5s × dozens of entries is fine, and the OAuth slice doesn't pool away). Add `config-set` shlex-parsing for the list-typed `mendeley_mcp_command` (no consumer asking; defaults are right for the user). Use a fuzzy title-match threshold for ambiguous detection (no real-data bug yet to calibrate against).
**Status:** Superseded — `sync-mendeley` action retired in Step 11.9. The `mendeley_client.py` MCP harness (`list_folders` / `list_documents`) survived; `lookup_doi` and `search_library` were removed. MCP SDK wiring principle (in-process SDK over hand-rolled JSON-RPC) still applies and is the basis for Step 13's MCP server adapter.
