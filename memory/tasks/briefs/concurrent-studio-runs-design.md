# Design: Concurrent Studio Runs (v1.3)

Generated by /office-hours on 2026-05-29
Branch: dev
Repo: Docent
Status: APPROVED
Mode: Builder

## Problem Statement

Today a user can only run one Studio action at a time. The frontend
`StudioRunProvider` holds a single run's state, and `startRun` calls `closeWs()`
before starting — so launching a second action *kills* the first. The user wants
to run multiple Studio actions concurrently (e.g. a deep research + a lit review +
a to-notebook), **inside one tab**, not by opening multiple browser tabs (which
`TabGuard` actively warns against).

## What Makes This Cool

You fire off three investigations, watch them stream side by side, and eventually
just submit a batch and walk away — coming back to a stack of finished briefs.
Docent stops being a one-thing-at-a-time tool and becomes a research desk.

## Constraints

- Concurrency must live **inside a single tab**. `TabGuard.tsx` (BroadcastChannel)
  warns when 2+ tabs are open; the design must not depend on multiple tabs.
- NotebookLM auth is a **single shared Playwright session, global to the machine**.
  Two `to-notebook` runs at once corrupt each other. The CLI can also launch a
  `to-notebook` run with no tab involved, so any mutex that only lives in the
  browser is insufficient.
- Free-tier aggregation (Tavily + Semantic Scholar + CrossRef) and Feynman credits
  are **account-global** — unbounded parallelism trips rate limits.
- `queue.json` has no file lock yet (already the #1 backlog item from the GLM review).
- Schema-safety rules apply to any new persisted job state (additive-only, defaults).

## Premises (all confirmed by user)

1. The engine is already concurrency-capable — each `/ws/studio/run` connection
   spawns its own subprocess (`opencode.py:316`). This is a manager-over-a-forking-
   engine job, not a backend rewrite.
2. Process isolation does NOT isolate shared external state (NLM session, account
   rate limits, output paths, `queue.json`). A resource-guard layer is mandatory.
3. "Auto-queue, just works" forces a real job model with state
   (queued/running/done/failed + reason), which is also exactly what the later
   fire-and-forget queue needs.
4. The hardening (file-lock + NLM mutex + rate-limit awareness) ships *with* this,
   not after. UI without it is a footgun.
5. Minimal-viable = "watch 2-3 in parallel"; the background queue is layer two.

### User clarification on premises
The intended UX is spawning concurrent actions **within one tab**, not opening
multiple tabs. The existing multi-tab path is discouraged by `TabGuard`, so the
design targets a single page holding N run-sockets.

## Decisions captured from the session

- **Core shape:** Both, phased — start with watch-in-parallel, design so a
  background queue layers on later without a rewrite.
- **Contention behaviour:** Auto-queue, just works — a run that can't safely start
  parks in a `queued` state with a human reason ("waiting: NotebookLM busy") and
  self-starts the moment the conflict clears. Zero clicks.

## Approaches Considered

### Approach A: Client-side run manager (minimal viable)
Refactor `StudioRunProvider` from one run to a `Map<runId, RunState>`, each with
its own WS. Output panel gets a run-switcher. No server changes.
- Effort: M · Risk: Med · Completeness: 6/10
- Con: NLM mutex advisory only (CLI can still collide); queue state dies with the
  tab; doesn't reach the fire-and-forget goal.

### Approach B: Server-side JobManager (ideal architecture)
`POST /api/studio/jobs` creates a job; server does admission control with real
resource locks (NLM mutex, rate-limit semaphore, `queue.json` file lock), spawns
the subprocess, streams per-job. Frontend is a thin view over the jobs API.
- Effort: XL · Risk: High · Completeness: 10/10
- Con: Touches CLI + server + frontend at once; overshoots a v1.3 timeline.

### Approach C: Phased hybrid (CHOSEN)
- **Slice 1 (v1.3.0):** Approach A's client run-manager + the non-negotiable
  server hardening only: `queue.json` file lock + a real **server-side NLM mutex**
  (machine-level lock the WS handler acquires before spawning any `to-notebook`
  subprocess). Auto-queue is client-mediated but backed by a real lock, so it's
  correct, not advisory. Ships "watch 2-3 in parallel" safely.
- **Slice 2 (v1.3.x / v1.4):** Promote the lock into a full server JobManager →
  true fire-and-forget queue + persistence. Slice 1's mutex is the seed of Slice
  2's resource manager — extension, not rework.
- Effort: Slice 1 = M-L · Risk: Med · Completeness: 8/10 now, 10/10 after Slice 2.

## Recommended Approach

**Approach C.** Honors both "both, phased" and "auto-queue just works" without
shipping a footgun (A) or blowing v1.3 scope (B). The one piece of A that A skips
and shouldn't — the server NLM mutex — is pulled into Slice 1.

## Implementation sketch (for /plan-eng-review, not built here)

### Slice 1 — frontend
- `studio-run-context.tsx`: replace singleton run state (`status`, `logs`,
  `sources`, `currentPhase`, `doneData`, `runRef`, `abortRef`, `stoppedRef`) with
  a keyed `Map<runId, RunState>`. Each run owns its WS + ref bundle.
- `startRun` must NOT `closeWs()` sibling runs — only the targeted run.
- Output panel (`_output.tsx`) + studio page: add an active-run switcher
  (list of running jobs, click to view its live log). Reuse `RunRecord` shape.
- `app-run-context.tsx`: represent N concurrent activities, not one 'studio' slot,
  so the Sidebar pill / StatusBanner reflect multiple runs.
- Client-side admission: before opening a `to-notebook` socket while another
  to-notebook is live, mark the new run `queued` with reason; start it on the
  prior one's `done`/`error`/`close`. Enforce a configurable parallel cap
  (default 3).

### Slice 1 — server (the mandatory hardening)
- Server-side NLM mutex: a machine-level lock (lockfile under the app data dir, or
  an `asyncio.Lock` + advisory file lock) acquired in the `/ws/studio/run` handler
  (`opencode.py`) before spawning a `to-notebook` subprocess; released on exit.
  Protects against CLI-vs-UI collision, which the client mutex cannot.
- `queue.json` file lock (already backlog #1) — apply during `reading_store`
  read/modify/write so concurrent runs can't corrupt the queue.
- Per-run output paths: ensure default output filenames are unique per run so two
  runs can't clobber one file.

### Slice 2 — server JobManager (later)
- `POST /api/studio/jobs` + job registry with persisted state
  (`queued/running/done/failed` + reason). Admission control with NLM mutex +
  rate-limit semaphore. Per-job stream endpoint. Frontend becomes a thin view.

## Open Questions

- Parallel cap default: proposed 3, configurable via a new `studio` config key.
  (Confirm key name + whether it's global or per-backend in /plan-eng-review.)
- Slice 1 NLM mutex mechanism: lockfile vs. asyncio.Lock+advisory lock — pick the
  one that behaves correctly on both Windows and WSL (Testing Rule applies).
- Rate-limit semaphore: Slice 1 relies on the parallel cap as a crude limiter;
  decide whether a per-provider semaphore is needed before Slice 2.

## Success Criteria

- Two non-NLM Studio actions (e.g. deep-research + lit) run and stream
  simultaneously in one tab; neither cancels the other.
- A second `to-notebook` launched while one is running shows `queued`
  ("waiting: NotebookLM busy") and auto-starts when the first finishes.
- A CLI `to-notebook` and a UI `to-notebook` cannot run at the same time (server
  mutex proven on Windows AND WSL).
- `queue.json` survives concurrent writes without corruption.
- No regression to single-run behaviour for users who only run one thing.
